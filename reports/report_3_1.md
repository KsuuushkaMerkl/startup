# Сервис микростажировок  
## Описание целевого сегмента и портрет пользователя

---

## Шаг 1. Уточнение выбранного сегмента

**Сегмент:**  
Мы рассматриваем сегмент **тимлидов / product-менеджеров в SMB IT-компаниях (10–200 сотрудников), работающих с короткими задачами разработки/аналитики/дизайна**, поскольку:
- у них регулярно возникают небольшие задачи, которые нецелесообразно отдавать в долгую стажировку или подряд;
- они быстрее принимают решения и готовы тестировать новые инструменты для оптимизации времени и затрат.

---

## Шаг 2. Сигналы

- Компании массово используют тестовые задания при найме → значит есть потребность в проверке навыков через реальные задачи  
- На фриланс-биржах часто жалуются на нестабильное качество исполнителей → проблема доверия и контроля качества  
- Вакансии джунов требуют коммерческий опыт → кандидаты не проходят фильтры  
- Тимлиды жалуются на перегруз и нехватку времени на менторинг стажеров  
- Хакатоны используются как инструмент поиска решений → компании готовы к конкуренции решений  
- В стартапах часто есть «висящие задачи», которые не приоритетны для команды  
- Кандидаты выполняют бесплатные тестовые → демотивация и низкая конверсия  
- Компании хотят быстрый результат, а не обучение → смещение к результат-ориентированному подходу  
- Популярность skill-based hiring → важнее результат, чем диплом  
- Фриланс используется как «костыль», а не идеальное решение → есть неудовлетворённый спрос  

---

## Шаг 3. Паспорт сегмента

- **Кто:**  
  Тимлид / Product Manager / Tech Lead в SMB IT-компании  

- **Контекст:**  
  Возникают небольшие задачи (фичи, прототипы, аналитика), которые не входят в приоритет спринта  

- **Цель:**  
  Быстро получить качественное решение задачи с минимальными затратами времени и ресурсов  

- **Триггер:**  
  - накопление задач в бэклоге  
  - нехватка ресурсов команды  
  - дедлайны или эксперименты  

- **Текущая замена:**  
  - фриланс  
  - тестовые задания кандидатам  
  - внутренняя команда (с задержкой)  

- **Потери:**  
  - время на поиск и проверку исполнителей  
  - деньги на неудачные попытки  
  - риск получить слабое решение  

- **Ограничения:**  
  - бюджет  
  - безопасность (NDA, код, данные)  
  - сроки  
  - качество  

- **Критерии выбора:**  
  - качество решения  
  - скорость получения результата  
  - минимальный риск  
  - прозрачность процесса  
  - цена  
  - удобство взаимодействия  

- **Каналы доступа:**  
  - LinkedIn  
  - Telegram-чаты (IT/стартапы)  
  - VC.ru, Habr  
  - профильные конференции  
  - cold outreach

## Шаг 4. Persona

### Персона: Алексей, Tech Lead

**Описание:**  
Алексей — техлид в продуктовой IT-компании (~80 человек). Отвечает за разработку и качество продукта. У него постоянно есть backlog задач, которые не доходят до реализации из-за приоритетов команды.

---

### Цели
- Быстро закрывать задачи без перегрузки команды  
- Получать качественные решения  
- Снижать риски при выборе исполнителей  
- Оптимизировать затраты  

---

### Боли / фрустрации
- Фрилансеры дают нестабильный результат  
- Много времени уходит на отбор  
- Стажеров нужно обучать → дорого  
- Бесплатные тестовые не дают мотивации кандидатам  
- Нет удобного способа сравнить несколько решений  

---

### Контекст
- Ограниченное время  
- Давление сроков  
- Ответственность за качество  
- Ограниченный бюджет  

---

### Текущий процесс
1. Ищет исполнителя (фриланс/знакомые)  
2. Даёт задачу  
3. Ждёт результат  
4. Часто переделывает или отклоняет  

---

### Критерии выбора (топ-5)
- Качество результата  
- Скорость  
- Надёжность  
- Цена  
- Простота процесса  

---

### Возражения / риски
- «Кто будет проверять качество?»  
- «А если украдут код?»  
- «Слишком сложно внедрить»  
- «Не будет нормальных кандидатов»  

---

### Где обитает
- Telegram (IT-чаты)  
- LinkedIn  
- Habr / VC.ru  
- GitHub / профессиональные комьюнити

## Шаг 6. Инсайты и гипотезы

### Инсайты
- Компании платят не за процесс, а за результат  
- Главная боль — не найти исполнителя, а выбрать качественного  
- Есть запрос на конкуренцию решений  
- Время тимлида — самый дорогой ресурс  
- Фриланс — компромисс, а не идеал  
- Кандидаты готовы работать, если есть оплата  
- Доверие и безопасность — ключевой фактор  

---

### Гипотезы

- H1: Для SMB тимлидов ключевая боль — риск получить некачественное решение, поэтому они готовы платить только за выбранный результат  

- H2: Если дать входной тест, компании будут доверять кандидатам больше  

- H3: Конкуренция решений повышает качество результата  

- H4: Тимлиды готовы использовать платформу, если она экономит время  

- H5: Кандидаты охотнее участвуют, если есть оплата  

- H6: Простота UX критична для внедрения  

- H7: Без гарантий безопасности компании не будут использовать сервис  

---

## ТОП-3 гипотезы + проверка

### H1 (самая важная)
- **Гипотеза:** компании готовы платить только за выбранное решение  
- **Проверка:** лендинг с CTA «создать задачу с депозитом»  
- **Метрика:** % компаний, внесших депозит (>10%)

---

### H2
- **Гипотеза:** входной тест повышает доверие  
- **Проверка:** интервью + прототип отбора  
- **Метрика:** субъективная оценка доверия (>70%)

---

### H3
- **Гипотеза:** несколько решений → выше качество  
- **Проверка:** пилот с 3–5 кандидатами на задачу  
- **Метрика:** % случаев, где выбирается лучший вариант (>60%)

---

## Итог

- Выбран сегмент: SMB IT (тимлиды/PM)  
- Ключевая боль: риск и время при выборе исполнителя  
- Основная ценность: **конкуренция решений + оплата за результат**  
- Подтверждение гипотез: через лендинг, интервью и пилоты
